软件需求分析设计

第二章

需求调研/获取：

需求获取的重要性：最关键、最困难、最易出错、最需要交流
需求获取的前提条件： 领域知识（业务知识）、沟通能力

需求：是产品必须完成的事（功能性需求）以及必须具备的品质

需要的东西：需求形式（样式风格）、数据信息、需求的内容

需求的种类：

  • 功能性需求：系统应该做的
  • 非功能性需求：质量+约束，系统特定的特性或约束

需求层次：

  • 业务需求
    • 反映了组织结构或客户队系统产品高层次的目标要求
    • 业务目标：系统建设目标
  • 用户需求
    • 描述了用户使用产品必须完成的任务
  • 功能需求（软件需求）
    • 定义开发人员必须实现的软件功能，使得用户能完成他们的任务，从而满足业务需求
  • 技术需求（具体的实现细节）

需求分析极致目的：所有人做的都一样

项目中标：

中标前：项目应标成功

  • 获取的要素：业务需求和用户需求 (全面性)
    中标后：确定项目范围
  • 要点：全部用户需求、部分功能需求

需求获取：

需求获取定义：

是通过客户、系统用户和其他与系统开发相关的人员交流发现系统需求的过程，是一个确定和理解不同用户种类的需要和限制的过程

需求获取总原则：

  • 先获取系统的总体目标（宏观）（业务需求）
  • 接着获取当前工作以及当前问题的信息（中观）（用户需求）
  • 然后系统应先处理的详细问题（微观）（用户需求）

需求获取第一步：

相关人员分析：是指那些直接或间接从开发的系统中受益的人、发现所有可能的需求源

需求获取前的准备：

明确问题清单（调研提纲）
+ 问题类型：封闭性问题、开放性问题

需求获取技术：

  • 访谈（领域知识、沟通能力）
  • 问卷调查（半封闭性问题：问卷设计（问题）、小范围验证）
  • 观察（演示）
  • 会议
    • 会议前：
      • 材料准备
      • 会议通知与会人员
    • 会议中：
      • 会议时间：1-1.5h
      • 设定会议主席：控制会议进程
      • 发言：2-3min
      • 记录
    • 会议后：
      • 会议纪要
      • （签字）
  • 文档研究：
    • 国家相关的法律法规
    • 行业规范
    • 企业（公司）规章制度
    • 岗位工作流程、职责、操作（工作）手册
    • 各类报表
    • 用户原始需求文档

需求获取

  • 第一步：需求确定（获取）
  • 原型：（目的：获取用户需求）
    • 抛弃型：纸和笔、画图工具/类型图片、原型开发工具
    • 进化型

需求背后的需求

  • 业务流程模型
  • 系统原型

需求获取的障碍

  • 遗漏
  • 错误

风险处理

  • 识别
  • 判别
  • 防范解决

系统环境模型（第一个模型）

  • 上下文图
  • 业务用例模型（图）

接口

  • 系统间接口（架构设计）
    • 通信协议
    • 数据内容和格式
    • 数据获取方式
    • 数据同步方式
  • 模块（组件）间接口（架构设计）
  • 类间接口（详细设计）

需求调研后的系统功能特征

  • 系统应该提供搜索功能
  • 系统必须具备分类浏览的功能

业务流程模型：带泳道的活动图

（主要通过泳道活动创建）

创建业务流程模型的目的

  • 对需求理解
  • 发现遗漏的需求

需求依赖矩阵：判断

  • 发现冲突
  • 提取公共组件

需求跟踪矩阵：需求分析的完整性

应用系统的灵魂：业务

需求分析

需求分析的主要工作

就是从“业务需求”“用户需求”分析得到“软件功能需求“的过程

需求分析的方法

  • 结构化分析方法：DFD、E-R
  • 面向对象方法：用例模型、概念模型

使用功能说明来表达需求的不足

  • 通常忽略其关联的用户及其为用户提供的利益
  • 不够精确，难以区分在不同场景下系统所标出的行为差异
  • 不容易表达出系统与其上下文之间的关系
  • 非功能性需求与功能性需求说明被人为割裂难以清晰表达两者之间的关系

用例建模

用例建模的作用和意义

是从外部用户和外围系统的角度，分析和考察待开发系统的行为，并通过用户与系统之间的交互系统描述了系统对外提供的功能特征

什么是用例模型

代表了从最终用户角度看的系统的功能和环境，是用户和开发者之间的契约

为什么要创建用例模型

  • 用例模型允许用户和系统开发者之间用一种用户可以理解的语言交流系统要做什么（软件开发者要做什么）
  • 是用户和系统开发者之间的可视化契约

什么是用例

  • 用户（参与者）与系统功能特性间的交互关系就是用例
  • 用户实例是在系统中执行的一系列动作，这些动作将生成持续参与者能看见的价值结果

用例建模技术最大限度的消除了需求中的不明确、不完整和不一致

用例建模流程

  1. 确定系统范围
    • 系统环境模型
    • 业务用例模型
  2. 识别参与者及其目标
  3. 基于参与者的识别
  4. 图例详述
  5. 分解、合并用例

用例规约

  • 用例名称
  • 用力概述
  • 参与者
  • 前置条件
  • 事件流（交互关系）
    • 基本流（正常）
    • 备选流（异常）
